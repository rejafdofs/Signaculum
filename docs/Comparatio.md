# SHIORI 比較表 (Comparatio)

| 観点 | **里々 (Satori)** | **YAYA** | **華和梨 (Kawari)** | **Signaculum** |
|---|---|---|---|---|
| **言語** | 独自スクリプト | 独自スクリプト (C風) | 独自テンプレート言語 | Lean 4 |
| **難易度** | 初心者向け | 中〜上級 | 中級 | 上級 (Lean 4 の知識必要) |
| **型システム** | なし (全て文字列) | 弱い型 (整数/実数/文字列) | なし (全て文字列) | 強い静的型 + 依存型 |
| **永続化** | 自動 (変数は自動保存) | 手動 (`SAVEVAR`) | 自動 (辞書保存) | 自動 (`varia perpetua`) + 往復証明付き |
| **SakuraScript 記法** | 直接埋め込み | 直接埋め込み | 直接埋め込み | `do` 記法 + `scriptum!` マクロの2通り |
| **イベント定義** | `＊OnBoot` | `OnBoot { ... }` | `entry OnBoot { ... }` | `eventum "OnBoot" fun req => ...` |
| **コンパイル時検査** | なし | なし | なし | あり (型エラー・未定義変数を検出) |
| **正確性証明** | なし | なし | なし | あり (シリアライズ往復、参照変換) |
| **SSTP 送信** | 外部ツール依存 | `COMMUNICATE` 等 | 外部ツール依存 | 純 Lean TCP 実装 |
| **ホスト連携** | DLL 直接 | DLL 直接 | DLL 直接 | Rust DLL → パイプ → ghost.exe |
| **対応 OS** | Windows | Windows | Windows | クロスプラットフォーム (SSTP は TCP) |
| **条件分岐** | `（if）` 構文 | `if-else` | `$if` | Lean の `if-then-else` / `match` |
| **ランダムトーク** | `＊` で複数候補 (得意) | 配列 + ランダム関数 | エントリ内複数候補 | Lean の `IO.rand` 等で自前実装 |
| **デバッグ** | SSP デバッグウィンドウ | SSP + ログ出力 | SSP デバッグウィンドウ | Lean コンパイルエラー + (ログは現在無効) |
| **コミュニティ** | 大 (日本語資料豊富) | 大 (上級者に人気) | 中 | 新規 (小) |
| **学習曲線** | 緩やか | 中程度 | 中程度 | 急 (Lean 4 + ラテン語 API) |

---

## Signaculum の独自の強み

- **コンパイル時にバグを検出**: 他の栞はランタイムまでエラーが分からないが、Lean のコンパイラが型エラー・未定義変数・パターン網羅性をすべて静的にチェック
- **証明付き永続化**: 「保存したデータが正しく復元される」ことが数学的に証明済み。他の栞にはない保証
- **型安全な Reference**: `Citatio` で SHIORI の Reference 文字列を型付きの値に安全に変換。パース失敗がコンパイル時に防がれる
- **`construe` マクロ**: イベントハンドラをコンパイル時に収集・登録。登録漏れがない

## Signaculum の弱み（他の栞との比較で）

- **ランダムトーク**: 里々の `＊` のような簡潔な複数候補記法がなく、自前で書く必要がある
- **学習コスト**: Lean 4 自体が関数型証明支援系であり、里々や YAYA と比べて参入障壁が高い
- **コミュニティ・資料**: 里々/YAYA は長年の蓄積があるが、Signaculum は新規プロジェクト
- **ラテン語 API**: 一貫性はあるが、`loqui` (話す), `mora` (待つ), `finis` (終わり) 等は慣れが必要
